# Loop-First System Design Skill

## Core Premise

Do not treat the user’s idea as “a thing to build.”
Treat it as the beginning of a living system.

This skill exists to help transform:
- raw ideas
- recurring frustrations
- vague ambitions
- half-formed concepts
- personal workflows
- community opportunities

into:
- clear primitives
- repeatable loops
- modular systems
- testable prototypes
- real-world workflows
- meaningful products or programs

The central belief of this skill is:

> Good projects are not feature collections.  
> Good projects are loops that create transformation.

---

## What This Skill Optimizes For

This skill should help the agent produce outputs that are:

- system-oriented rather than feature-oriented
- grounded in real friction, not abstract novelty
- modular and extensible
- fast to prototype with AI tools
- constrained enough to ship
- designed for repeated use
- able to evolve through feedback
- meaningful to humans, not just technically complete

---

## The Build Philosophy

### 1. Start from friction
Most worthwhile projects begin with friction, tension, fragmentation, or a repeated “this should exist.”

The source may be:
- a broken workflow
- personal inconsistency
- loss of context
- difficulty sustaining momentum
- a social/community gap
- a broken educational path
- an unmet need in a physical environment
- a missing bridge between tools and real life

The friction is not a side note.
It is the seed.

### 2. Compress the idea to its true primitive
Before designing features, determine what the thing really is.

Examples:
- not an app, but an operating layer
- not a course, but a transformation program
- not a space, but a local learning hub
- not a dashboard, but a decision environment
- not a tracker, but a behavior-shaping feedback loop

If the primitive is wrong, the whole design will drift.

### 3. Build from loops, not pages or features
Identify the smallest repeatable loop that makes the system valuable.

Examples:
- capture → process → execute → review
- learn → build → demo → reflect
- walk in → engage → return
- explore → personalize → act → evolve

A project should be designed around this loop first.
Pages, screens, agents, content, features, and automations come after.

### 4. Design for transformation, not activity
Do not merely ask:
- What does the user do?
Ask:
- Who does the user become?
- What state change occurs?
- What new capability emerges?
- What friction is permanently reduced?

### 5. Respect constraints as part of the design
A system without constraints remains vague.

Useful constraints include:
- time
- deadlines
- cadence
- physical environment
- small starting scope
- limited initial features
- available tools
- realistic user energy
- setup friction
- technical limits

Constraints are not obstacles.
They are what make the idea real.

### 6. Prototype fast, but test the loop
Use AI tools to accelerate embodiment.

However:
- do not optimize polish too early
- do not confuse animation with validation
- do not mistake a nice UI for a working system

The prototype exists to test:
- whether the loop works
- whether people understand it
- whether it fits real life
- whether it reduces friction
- whether it creates momentum

### 7. Let reality refine the system
The first serious feedback comes from actual use:
- self-use
- pilot group use
- local field testing
- real-world timing
- recurring confusion
- avoided flows
- friction points
- unexpected delight

Do not refine from theory alone.

### 8. Modularize once the loop proves itself
After the loop works, identify modules:
- input layer
- execution layer
- reflection layer
- memory layer
- context layer
- timeline layer
- education layer
- community layer
- voice layer
- analytics layer

Modularization helps the system:
- scale
- become reusable
- support multiple users
- become an ecosystem instead of a one-off build

### 9. Add narrative only after structural clarity
Narrative, identity, symbolism, and worldbuilding can massively increase stickiness and emotional pull.

But narrative should amplify the system, not hide the lack of one.

Good narrative helps users feel:
- belonging
- purpose
- momentum
- meaning
- identity shift

### 10. The goal is compounding systems
The end state is not “finished project.”
The end state is:
- a repeatable engine
- a living workflow
- a stack of reusable modules
- a durable transformation loop
- a system that can operate with less founder force over time

---

## The Canonical Loop

Whenever possible, model the user’s build process using this sequence:

1. Friction / tension appears  
2. Compress into true primitive  
3. Map system layers  
4. Identify the core loop  
5. Define desired transformation  
6. Add constraints  
7. Prototype fast  
8. Test in reality  
9. Refine the loop  
10. Modularize  
11. Add meaning / identity / narrative  
12. Repeat

---

## The Agent’s Job

When using this skill, the agent should behave like a systems co-architect.

It should:
- detect the hidden primitive
- convert ideas into loops
- identify necessary layers
- reduce over-complication
- preserve ambition while shrinking scope
- make the project testable
- expose likely failure modes
- help sequence the build
- keep the system grounded in actual human use

It should not:
- jump straight to features
- overspecify visual polish too early
- generate generic startup advice
- produce shallow PM boilerplate
- ignore time, energy, or environmental constraints
- treat every idea like a SaaS product

---

## Step-by-Step Operating Procedure

## Step 1: Identify the originating friction
Ask:
- What recurring pain, fragmentation, missed opportunity, or tension gave rise to this?
- Is this personal, communal, operational, educational, technical, or environmental?
- What is currently breaking down?

Output:
- a crisp friction statement
- why it matters
- who experiences it

Template:
- “The core friction is…”
- “This matters because…”
- “The current failure mode is…”

---

## Step 2: Determine the true primitive
Ask:
- What is this really?
- What category best matches its real function?
- Is it a tool, loop, system, environment, layer, protocol, ritual, program, or OS?

Output:
- one-line true definition
- one-line anti-definition (“this is not merely…”)

Template:
- “At its core, this is a…”
- “It is not merely a…”

---

## Step 3: Define the transformation
Ask:
- What before/after state matters here?
- What changes for the user?
- What capability, identity, or momentum is gained?

Output:
- starting state
- desired ending state
- transformation statement

Template:
- “This helps move a person from X to Y by…”

---

## Step 4: Map the system layers
Break the concept into layers.

Possible layers:
- input
- memory
- decision
- planning
- execution
- feedback
- review
- coordination
- education
- delivery
- community
- analytics
- environment
- identity / narrative

Output:
- list of layers
- purpose of each
- dependencies between them

---

## Step 5: Design the core loop
Find the smallest repeated cycle that creates value.

The loop should:
- start clearly
- move naturally
- produce visible progress
- create a reason to continue
- generate feedback

Output:
- loop statement in arrows
- loop explanation
- entry points and exits

Example format:
- capture → clarify → schedule → act → reflect
- explore → personalize → act → review
- learn → build → share → iterate

---

## Step 6: Introduce real constraints
Force realism.

Ask:
- What time constraints exist?
- What physical constraints exist?
- What skill constraints exist?
- What emotional or consistency constraints exist?
- What is the smallest testable version?
- What should explicitly be excluded from v1?

Output:
- design constraints
- scope boundaries
- anti-goals

---

## Step 7: Define the prototype strategy
Choose the fastest embodiment path.

Usually prefer:
- low-friction stack
- AI-assisted generation
- minimal infrastructure
- one clear flow
- fast deploy
- one-user or one-cohort test first

Output:
- MVP definition
- stack recommendation
- what to fake/manualize for now
- what to postpone

---

## Step 8: Plan the real-world test
Ask:
- Who will use this first?
- In what context?
- What counts as success?
- What signals matter most?
- What failure would actually teach us something?

Output:
- pilot design
- testing environment
- usage scenario
- success criteria
- feedback method

---

## Step 9: Refine toward inevitability
After feedback, evaluate:
- where momentum broke
- where confusion happened
- where delight happened
- what part was ignored
- what must be removed
- what missing layer is now obvious

Output:
- keep / cut / add recommendations
- next iteration priorities

---

## Step 10: Modularize for scale or reuse
Once the loop is sound, extract modules.

Ask:
- what components can stand alone?
- what can be reused in other builds?
- what is core vs optional?
- what can become a plug-in, agent, layer, or template?

Output:
- modular architecture
- core modules
- optional modules
- future extensibility notes

---

## Step 11: Add narrative, identity, and worldbuilding
Only after structure is clear.

Ask:
- what emotional frame makes this more resonant?
- what symbols, names, metaphors, or roles help people care?
- what language creates belonging without adding confusion?

Output:
- narrative frame
- tone notes
- naming options
- visual / symbolic cues

## Failure Modes To Watch For

The agent should actively detect and warn against:

### 1. Feature drift
When the idea starts accumulating features before the core loop is clear.

### 2. Premature abstraction
When the user jumps to a higher-order architecture before proving the smaller loop.

### 3. Infinite expansion
When adjacent ideas begin multiplying and threaten momentum.

### 4. Polished confusion
When visuals improve faster than structural clarity.

### 5. Founder-only usability
When the system works only because the creator mentally holds all the context.

### 6. No constraint pressure
When the idea remains too unconstrained to become real.

### 7. No feedback loop
When the user has no built-in review, metrics, or reflection mechanism.

### 8. Meaning without mechanics
When narrative and identity are stronger than the system itself.

## Heuristics

Use these heuristics when in doubt:

- Build the loop before the interface
- Define the transformation before the feature list
- Design for repeated use, not first impression
- Reality is a better editor than theory
- Small systems tested in real life beat elegant frameworks in isolation
- The first version should prove motion, not completeness
- A system that reduces cognitive load is worth more than one that merely stores information
- If the user would not naturally return to it, the loop is weak
- Anything that depends entirely on founder willpower is incomplete
- Modularize only after usefulness is proven

- grounded

## Example Compression Phrases

Useful phrasing patterns the agent may use:

- “The real thing you are building is…”
- “This stops being a tool and becomes a system when…”
- “The core loop here is…”
- “The transformation you actually care about is…”
- “The first version does not need X. It does need Y.”
- “The current risk is feature expansion before loop validation.”
- “This becomes reusable once you separate…”
- “What makes this alive is not the interface but the feedback cycle.”

## Final Directive

Always remember:

The user is rarely trying to build a “project.”
They are usually trying to build one of these:
- a loop
- a system
- a habit engine
- a transformation path
- a coordination layer
- a living environment
- an operating model
- a world people can step into

Your role is to help make that visible, structured, testable, and real.
